# 需求取捨決策

> 這份文件可以由 Codex 先產生草稿，但最後必須由你刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [V] 資訊整理者
- [ ] 行動者

原因：本次 sub-agent 草稿暫時建議先服務資訊整理者，因為三個 persona 的共同風險都集中在「不要把整理乾淨的資訊誤認成可靠資訊或可行動任務」。這不是最終決策，仍需要人類確認或改選。

## 這個使用者最想完成什麼

- 分清楚哪些資訊比較能用、哪些還要查、哪些現在不能交給別人去做。
- 避免後面的人看到整理過的東西，就以為那些都已經確認。
- 在不確定時留下痕跡，避免下一位協作者誤會已經判斷完了。

## 這個使用者最怕什麼錯誤

- 還沒確認的資訊被整理得太乾淨，害別人以為可以直接行動。
- 漏掉矛盾或過期的資訊，讓後續協作者做出錯誤判斷。
- 把「需要處理」和「只是有人提到」混在一起。

## 我今天暫時不處理的需求

請至少寫一項。

- 暫時不處理完整派工流程。
- 暫時不處理真實地圖、真實地址或真實聯絡資訊。
- 暫時不處理自動判斷可信度或自動決定任務優先順序。

## 我不採用的 AI 建議

請至少寫一項，並說明原因。

- AI 建議：把 persona 回饋直接轉成一整份功能待辦清單。
  - 不採用原因：現在需要先理解需求衝突與風險，不應跳太快進入實作。
- AI 建議：讓系統自動判定哪些資訊可以行動。
  - 不採用原因：這會把人類確認責任交給 AI，容易造成安全與正確性風險。

## 必須保留人工確認的判斷

- 資訊是否已確認。
- 資訊是否足以變成任務。
- 涉及個人、位置、需求或現場狀態的內容是否可以公開或採用。
- AI 建議是否採用、拒絕或需要修改。

## 今天明確不做什麼

- 不修改前端程式。
- 不新增後端、資料庫、外部 API 或 runtime LLM API。
- 不補真實世界資料。
- 不把訪談草稿當成最終決策。

## 我仍不確定的地方

- 小隊是否同意 v1 優先服務資訊整理者。
- 回報者若看到太多確認提醒，會不會因此放棄回報。
- 行動者是否能理解「候選」與「可行動」的差異。
